We Can Fix It!

(And it doesn’t mean starting from scratch.)

The Solution:

Progressive Web Apps

With Progressive Web Apps (PWAs), we can:

  1. Significantly decrease the amount of data transferred on each page load
  2. Improve the experience for users on unreliable connections

Booooooooooooooring!

“That’s not my target market.”

PWAs seem like a lot of work to help users that may not be in your demographic. And a lot of the language around PWAs mentions international markets with flaky internet and poor infrastructure, so that’s not really an unfair thing to believe.
The checklist to meet “Progressive Web App” standards isn’t short, and a lot of the items on the list seem pretty complicated to implement. Adding PWA features to an app might sound an awful lot like being told to eat your vegetables.

Can’t we just ignore PWAs?

Nope

Here’s why:

  • PWAs decrease load times on all connections
  • Perceived load times improve
  • Stability improves for mobile users everywhere
  • User experience is better all around
By caching resources, repeat visists and navigating to new pages is lightweight — and it’s fast. It eliminates HTTP requests, so stability improves. And with all that combined, the user experience gets better — even for users on high-speed connections.

That sounds significantly more awesome than eating vegetables.

To build PWA we need to:

  • Register a Service Worker to manage caching and offline access
  • Create an app manifest to describe the app
  • Let the Service Worker know about required assets
  • Define caching rules for other assets

Bonus points for:

  • Lazy loading resources to improve perceived performance
  • Enabling offline support for user interactions

...that sounds like a lot of work.

It’s not

Most of the process is already done for you

Step 1: Install
sw-precache-webpack-plugin

  • Uses the excellent sw-precache tool from the Google Chrome team
  • Creates the Service Worker
  • Accepts a list of files or folders and generates required asset list
  • Accepts a list of URL patterns to manage caching of other assets
  • Enables offline access

Configuring sw-precache-webpack-plugin:

plugins: [
  new SWPrecacheWebpackPlugin(
    {
      filepath: './public/service-worker.js',
      runtimeCaching: [{
        urlPattern: /[.]jpg$/,
        handler: 'cacheFirst'
      }],
      staticFileGlobs: ['public/**.*'],
      stripPrefix: 'public/',
    }
  ),
],

github.com/goldhand/sw-precache-webpack-plugin

Step 2: Register the Service Worker in your app

<script>
  (function() {
    if('serviceWorker' in navigator) {
      navigator.serviceWorker.register('./service-worker.js');
    }
  })();
</script>

Step 3: Generate a manifest

  1. Go to realfavicongenerator.net
  2. Follow the instructions
  3. Done!

NOTE: We can automate this with an NPM package.

But what if we need a database?

There are 2 main flavors of databases

  • Server-side: the app connects to a remote machine to perform CRUD operations
  • Client-side: the database is on the same device as the app

The Low-Down on
Server-Side Databases

The Good

  • Accessible to all connected instances
  • Apps don’t need to load all data to run the app

The Bad

  • Can cause a noticeable slow-down in apps
  • Apps require an internet connection

The Low-Down on
Client-Side Databases

The Good

  • Apps feel much faster because there’s no server request
  • Even if there’s no internet connection, apps will still work

The Bad

  • Each app has its own instance of the database
  • All of the app’s data needs to be downloaded

What if I want the best of both databases?

Use ’Em Both

and sync that shit

CouchDB + PouchDB

With Replication

PouchDB gives us:

  • Offline-first capabilities
  • Near-instant updates and responses
  • A simple JavaScript API for CRUD operations

CouchDB gives us:

  • Data persistence across devices
  • Support for collaborative editing

It’s not hard

It’s pretty easy, actually.

// Keep the local (PouchDB) and the remote (CouchDB) databases in sync.
PouchDB.sync('local_db', 'https://example.com/remote_db', {
  live: true,
  retry: true,
});

// Whenever there are changes on the remote database, we want to know.
db.changes({
  since: 'now',
  live: true,
}).on('change', handleDataChanges);

Progressive Web Apps:

  • Improve the user experience on all devices
