How Do Databases Even Work, Anyways?

The Short Version

Databases at a Glance

  • A database is a collection of stuff (“documents”)
  • Credentials are created with varying permissions
  • App(s) will connect to the database as a user
  • The app can store, ask for (“query”), and delete stuff...
  • ...if its credentials have the right permissions

The Benefits of Databases

  • Databases are persistent. Data isn’t lost if the app restarts or the page is reloaded.
  • Databases are shared. Data changed by one user can be accessed by other users.
  • Databases are on-demand. Data is loaded as-needed, instead of needing everything in memory.

The Big 4

How We (Usually) Work With Databases

There are 4 primary database interactions:

  • Create. Adding a new document to the database. Often called an “insert”.
  • Read. Retrieving an existing document or documents from the database.
  • Update. Making changes to an existing document in the database.
  • Delete. Remove a document from the database.

You’ll hear these referred to as

CRUD

operations

Okay, so... where is all this data?

There’s nothing special about how databases are built. Typically, they’re a collection of folders, and the database itself is a software program to dig through the folders for the thing you’re looking for. Think of it like a really fast librarian.

There are 2 main flavors of databases

  • Server-side: the app connects to a remote machine to perform CRUD operations
  • Client-side: the database is on the same device as the app

The Low-Down on
Server-Side Databases

The Good

  • Accessible to all connected instances
  • Apps don’t need to load all data to run the app

The Bad

  • Can cause a noticeable slow-down in apps
  • Apps require an internet connection

The Low-Down on
Client-Side Databases

The Good

  • Apps feel much faster because there’s no server request
  • Even if there’s no internet connection, apps will still work

The Bad

  • Each app has its own instance of the database
  • All of the app’s data needs to be downloaded

CouchDB + PouchDB

With Replication

PouchDB gives us:

  • Offline-first capabilities
  • Near-instant updates and responses
  • A simple JavaScript API for CRUD operations

CouchDB gives us:

  • Data persistence across devices
  • Support for collaborative editing
  • A way to tie IBM products into the demo

It’s not hard

It’s pretty easy, actually.

// Keep the local (PouchDB) and the remote (CouchDB) databases in sync.
PouchDB.sync('local_db', 'https://example.com/remote_db', {
  live: true,
  retry: true,
});

// Whenever there are changes on the remote database, we want to know.
db.changes({
  since: 'now',
  live: true,
}).on('change', handleDataChanges);
